# Contributing to Open Manuscript Initiative

First of all, thank you for your interest in contributing to the **Open Manuscript Initiative (OMI)**.

OMI is more than a software project. It is an open initiative to improve the way scholarly manuscripts are created, reviewed, published, and preserved.

We welcome contributions from developers, researchers, editors, librarians, designers, translators, publishers, and anyone interested in open scholarly communication.

---

# Ways to Contribute

You do not have to write code to contribute.

We welcome contributions in many forms:

- Software development
- Bug reports
- Feature requests
- Documentation
- User interface design
- Accessibility improvements
- Internationalization and translations
- Scholarly publishing workflows
- Metadata standards
- Citation systems
- Testing
- Performance improvements
- Security reviews

---

# Before You Start

Before starting significant work, please:

1. Search existing Issues.
2. Search existing Discussions.
3. Open a Discussion if you would like to propose a new feature or major architectural change.

This helps avoid duplicated work and encourages collaborative design.

---

# Development Principles

OMI follows several core principles.

## Platform Independence

Features should not depend on a single publishing platform unless implemented as optional integrations.

## Open Standards

Whenever possible, use open standards such as:

- JATS
- CSL
- DOI
- ORCID
- Crossref
- DataCite
- Dublin Core
- Schema.org

## Modularity

New functionality should preferably be implemented as independent modules or plugins.

## Backward Compatibility

Breaking changes should be avoided whenever possible.

---

# Coding Guidelines

Please:

- write clean and readable code;
- document public APIs;
- include tests whenever practical;
- keep pull requests focused;
- explain the rationale behind significant changes.

---

# Documentation

Documentation is considered a first-class contribution.

If a feature is difficult to explain, it probably needs to be simplified.

---

# Issues

When opening an issue, please include:

- a clear description;
- steps to reproduce (if applicable);
- expected behaviour;
- actual behaviour;
- screenshots if helpful;
- environment information.

---

# Pull Requests

Before submitting a pull request:

- ensure the project builds successfully;
- run available tests;
- update documentation if necessary;
- keep commits logically organized.

Pull requests should describe:

- what changed;
- why it changed;
- whether it introduces breaking changes.

---

# Code Review

Reviews focus on:

- correctness;
- maintainability;
- usability;
- accessibility;
- interoperability;
- scholarly workflow.

Reviews are collaborative, not adversarial.

---

# Community

Please follow our Code of Conduct.

Respectful discussion and constructive feedback are essential to building a healthy open-source community.
